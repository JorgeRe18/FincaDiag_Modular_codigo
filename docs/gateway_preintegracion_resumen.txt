TAREA 1: CONGELAR EL CONTRATO DE SALIDA
----------------------------------------

Que significa:
  Definir exactamente que mensajes produce el gateway y que campos lleva cada
  uno. Esto es como un "acuerdo" de formato para que cualquier sistema que
  reciba esos mensajes sepa que esperar.

Que se hizo:
  Se revisaron los archivos .readable.json de dos sesiones de referencia:
  - 06/04/2026 PM 1PM (sesion combinada principal, 25 mensajes)
  - 09/04/2026 PM 1PM (sesion combinada secundaria, 19 mensajes)

  Se identificaron 8 tipos de mensaje:
  1. session_summary    - resumen general de la sesion
  2. baseline_snapshot  - metricas de red base (latencia, jitter, perdida)
  3. pcap_summary       - resumen de la captura de red
  4. alerts_summary     - alertas detectadas por severidad y capa
  5. collar_summary     - resumen de eventos del collar
  6. correlation_summary - correlacion entre serial y collar
  7. field_validation_summary - validacion presencial de campo
  8. cow_event          - evento individual por vaca (puede haber 0 o muchos)

  Para cada tipo se definieron los campos minimos obligatorios.

Resultado:
  Contrato v1 congelado y guardado en memoria para uso futuro.


TAREA 2: PRUEBA MQTT-TLS CONTROLADA
------------------------------------

Que significa:
  Demostrar que el gateway puede publicar mensajes a un broker MQTT usando
  conexion segura (TLS = cifrado, como HTTPS pero para MQTT).

Problema encontrado:
  No existia un broker configurado ni certificados en el proyecto.
  Tampoco habia un "responsable del broker" identificado.

Solucion:
  Se monto un broker de laboratorio en la laptop del usuario.

Pasos que se hicieron:
  1. Se instalo Mosquitto (broker MQTT) con winget
  2. Se instalo OpenSSL (herramienta para generar certificados) con winget
  3. Se crearon carpetas en C:\mqtt-lab\
  4. Se generaron certificados:
     - CA (autoridad certificadora local, como un "notario" digital)
     - Certificado del servidor (identidad del broker)
     - Certificado del cliente (identidad del gateway)
  5. Se creo un archivo de configuracion mosquitto.conf que:
     - Escucha en puerto 8883 (puerto estandar MQTT seguro)
     - Exige certificado de cliente (mTLS = ambos lados se autentican)
  6. Se levanto el broker
  7. Se probo con mosquitto_sub y mosquitto_pub que la conexion TLS funcionara
  8. Se ejecuto el gateway en modo mqtt-tls contra ese broker

Resultado:
  published=25, spooled=0, failed=0
  El gateway publico correctamente los 25 mensajes de la sesion principal.

Donde quedo todo:
  - Broker y certs: C:\mqtt-lab\
  - Certificados:   C:\mqtt-lab\certs\
  - Config broker:  C:\mqtt-lab\config\mosquitto.conf
  - Logs broker:    C:\mqtt-lab\log\

Variables de entorno usadas:
  MQTT_HOST=localhost
  MQTT_PORT=8883
  TOPIC_ROOT=fincadiag/la_esmeralda
  CA_PATH=C:\mqtt-lab\certs\ca.crt
  CERT_PATH=C:\mqtt-lab\certs\client.crt
  KEY_PATH=C:\mqtt-lab\certs\client.key
  TLS_MIN_VERSION=1.3


TAREA 3: VALIDAR RESILIENCIA ANTE FALLO DEL BROKER
---------------------------------------------------

Que significa:
  Verificar que pasa cuando el broker esta caido (apagado, no disponible).
  El gateway deberia guardar los mensajes localmente (spool) en vez de perderlos.

Que se encontro ANTES del arreglo:
  El gateway explotaba con un error:
    ConnectionRefusedError: [WinError 10061]
  Los mensajes se perdian y no se guardaban en ningun lado.

Que se arreglo:
  Se modifico src/fincadiag/gateway/runtime.py para que:
  - Si la publicacion falla (por cualquier razon), el batch se guarde en spool
  - El gateway termine limpiamente reportando failed=N en vez de crashear

Que se verifico DESPUES del arreglo:
  Con broker apagado:
    PUBLISHER= PahoPublisher
    published= 0 spooled= 0 failed= 25
    notes= ['Publicacion fallida, batch enviado a spool: [WinError 10061]...']
  Y el archivo .jsonl aparecio en la carpeta de spool.

Resultado:
  El gateway ya no crashea ante broker caido.
  Los mensajes se preservan localmente para reenvio futuro.

================================================================================
 CONCEPTOS CLAVE PARA ENTENDER LO QUE SE HIZO
================================================================================

QUE ES UN BROKER MQTT:
  Es un servidor intermediario. El gateway le "publica" mensajes, y otros
  sistemas se "suscriben" para recibirlos. Piensa en el como un buzón central.

QUE ES TLS:
  Transport Layer Security. Es cifrado de la conexion, como HTTPS pero para
  MQTT. Garantiza que nadie puede leer los mensajes en transito.

QUE ES mTLS (mutual TLS):
  Ambos lados (cliente y servidor) se autentican mutuamente con certificados.
  No basta con que el broker tenga cert; el cliente tambien debe presentar uno.

QUE ES UN CERTIFICADO:
  Un archivo digital que identifica a una entidad (servidor o cliente).
  Se firma con una CA (autoridad certificadora). En nuestro caso la CA es local.

QUE ES EL SPOOL:
  Una carpeta local donde el gateway guarda mensajes que no pudo publicar.
  Funciona como respaldo. La idea es que cuando vuelva el broker, se reenvien.

QUE ES dry-run:
  Un modo de prueba donde el gateway hace todo el procesamiento pero en vez de
  publicar por MQTT, guarda los mensajes en archivos locales (.jsonl y
  .readable.json). Sirve para validar sin tocar nada real.

QUE ES EL CONTRATO v1:
  Un acuerdo de que tipos de mensaje y que campos minimos debe tener cada
  publicacion del gateway. Si algo cambia despues, se trata como "version nueva".
